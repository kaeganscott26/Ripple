# RIPPLE GAME SCOPE

_Status: Master Scope | Codex Context | Game Boundary_

This document defines the intended shape of Ripple before code generation begins.

Ripple is a standalone game project.

INTERVENTION is source-code ARG material only.

Ripple uses the copied source material as symbolic fuel, not as a fixed script.

---

## One-Sentence Pitch

Ripple is a turn-based 2D living board game where players seed memories, observe AI-driven agents, inject small ripples, and watch those ripples become belief, law, myth, and nested worlds.

---

## Core Identity

Ripple should feel like:

- a Civilization-style strategy board
- a Jumanji-style living board
- an ARG archive becoming playable
- an agent simulation
- a recursive thought experiment about observation and consequence

Ripple should not feel like:

- a direct chapter adaptation
- a fixed visual novel
- a Minecraft clone
- a first-person horror game
- a lecture about theory

The game should demonstrate its ideas through systems.

---

## Player Role

The player is the Observer.

The Observer does not directly control every agent.

The Observer can:

- choose the opening memory mode
- seed agent memories
- inspect rooms
- inspect artifacts
- inspect agents
- inject ripples
- advance turns
- watch social reality form
- watch institutions emerge
- unlock nested simulations
- export simulation logs as artifacts

The Observer has access to more information than the agents, but should not feel fully omniscient.

The Observer changes conditions.

The board interprets consequences.

---

## Opening Modes

Ripple begins with a mode selection screen.

### Mystery

The board chooses every agent's remembered life secretly.

The player discovers who the agents are by watching behavior.

### Vague

The player chooses A, B, or C for each agent without seeing the full life history.

The player gets control without full knowledge.

### Experimental

The player sees what each memory seed means before choosing.

This mode supports testing, replay, and intentional world design.

---

## Memory Seed Rule

Characters do not have one fixed playable past.

Each agent has:

- a stable identity core
- Life A
- Life B
- Life C
- possible fragments
- runtime memories generated by the current playthrough

At the beginning of a run, the game determines which life each agent remembers.

The agent believes that remembered life was always theirs.

The player may not know which life is active unless the selected mode reveals it.

---

## Room Rule

Chapters are not levels.

Chapters are symbolic source material.

Ripple converts source chapters into room functions.

A room is defined by:

- symbolic function
- core artifact
- pressure type
- active agents
- possible ripple behaviors
- reality-layer risk
- unlock paths
- emergent outcomes

The room must not replay the source chapter beat-for-beat.

The room should generate new outcomes from the symbolism.

---

## Reality Layers

Every major event should be tracked through four layers.

### Base Reality

What actually happened inside the simulation engine.

### Perceived Reality

What individual agents believe happened.

### Social Reality

What the group begins repeating, accepting, resisting, or debating.

### Institutional Reality

What becomes a rule, law, ritual, role, archive, taboo, doctrine, or inherited structure.

The game should make these layers visible through UI panels and event logs.

---

## Turn Loop

A standard turn should follow this shape:

1. Observation Phase
2. Ripple Phase
3. Agent Phase
4. Board Phase
5. Reality Layer Phase
6. Archive Phase

The main turn button may be called:

- Advance Ripple
- Render Next Turn
- Let the Board Respond

---

## Agents

Agents are characters or symbolic entities built from the source material.

They should not be generic NPCs.

Agents need:

- identity core
- memory variant
- beliefs
- fears
- desires
- relationships
- triggers
- voice pattern
- ripple behavior
- observer suspicion
- inheritance rules

Agents should not be described as literally conscious.

Use stronger framing:

Agents develop self-modeling behavior, world-modeling behavior, and observer-directed beliefs over time.

---

## Artifacts

Artifacts are not normal collectibles.

Artifacts are active symbolic objects.

An artifact can:

- change a room's pressure
- trigger memories
- alter agent interpretation
- become law
- become myth
- become an inherited object inside a nested simulation

Example:

The Boulder is not a level prop.

The Boulder is a consequence object.

It may become a joke, obstacle, law, warning, origin myth, or ritual depending on how the board remembers it.

---

## Nested Simulation

Late-game Ripple includes recursive simulation.

Agents may eventually build their own simulation inside the simulation.

They do not have access to the full source material.

They can only pass down what they observed, remembered, misunderstood, feared, preserved, distorted, or institutionalized.

Core law:

**No layer can transmit what it cannot observe.**

This creates origin drift.

A source artifact may become a belief.

A belief may become a myth.

A myth may become law.

A law may become physics in a deeper layer.

## First Demo Target

The first playable demo should be small.

Recommended target:

**Ripple: The Boulder Build**

Include:

- small 2D board map
- Boulder Room
- a few linked rooms
- core agent cards
- artifact panel
- memory seed selection
- turn loop
- ripple input
- event feed
- reality layer panel
- exportable run log

The first demo proves consequence.

Not scale.

---

## Technical Direction

The game should be local-first.

Initial version:

- lightweight 2D board UI
- local files for data
- turn-based simulation
- deterministic rules before AI
- optional local AI later
- no paid API required for core play

Core rule:

**Code handles mechanics. AI handles meaning.**

Use AI only when interpretation, dialogue, reflection, or belief mutation is needed.

---

## Codex Boundary

Codex should not start by building a full game.

Codex should first:

1. read this scope
2. read the context docs
3. extract agent and room structures
4. preserve source material
5. create data schemas
6. build a small rules-based prototype
7. add AI only after the skeleton works

Do not let Codex creatively rewrite canon unless explicitly instructed.

Codex is the builder.

The repo is the source material.

The game is the living board.
